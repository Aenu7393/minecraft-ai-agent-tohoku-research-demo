# Minecraft AI Agent Research Demo

## 1. 概要

このリポジトリは、Minecraft内で動作する自律AIエージェントに関する研究プロジェクトの公開用デモ版です。

本研究の目的は、エージェントが現在のゲーム状況を理解し、高レベルな目標を実行可能なサブゴールへ分解し、低レベル行動を実行し、経験を記憶し、失敗から回復できる仕組みを構築することです。

特に本プロジェクトでは、LLMを単に計画生成に用いるだけでなく、VLMによる視覚的状態理解、人間の記憶システムに着想を得た記憶アーキテクチャ、そして低レベル行動の実行中に認知的な監視・補正・中断・再計画を行う仕組みに重点を置いています。

さらに、本研究ではLLMが低レベル行動を直接すべて操作するのではなく、状況に応じて低レベル方策の行動分布や選択傾向を変調し、その結果を経験として蓄積して低レベル方策を再学習する階層型エージェントを目指しています。

通常時は、LLMが高レベル目標をサブゴールへ分解し、強化学習や模倣学習などで獲得した低レベル方策が具体的な移動・採掘・クラフト・攻撃などを実行します。一方で、失敗が発生した場合や、繊細な操作・状況判断が必要な場合には、LLM/VLMなどの思考モジュールがサブゴール生成だけでなく、低レベル方策の実行過程にも介入します。

この介入によって、低レベル方策の行動選択を一時的に補正し、成功または失敗の結果を記憶として保存します。そして、その経験を用いて低レベル処理モジュールを再学習することで、エージェントが次回以降より自律的に適切な行動を取れるようにすることを目指しています。

---

## 2. 研究の背景

大規模言語モデルは、高レベルなタスク計画を生成することができます。例えば、「石のツルハシを作る」という目標に対して、「木を集める」「板材を作る」「棒を作る」「木のツルハシを作る」「丸石を掘る」といったサブゴール列を生成できます。

しかし、実際のMinecraft環境でタスクを達成するためには、単にサブゴール列を生成するだけでは不十分です。エージェントは、現在の所持アイテム、周囲のブロック、視覚的状況、敵や障害物の有無、サブゴールの実行可能性、失敗理由、過去の経験などを考慮する必要があります。

また、サブゴール自体が正しくても、低レベル行動の実行がうまくいかない場合があります。例えば、木を切るというサブゴールが与えられていても、対象の木を見失ったり、視点がずれていたり、障害物に引っかかったり、危険な敵が近くにいたりする場合があります。

このような状況では、LLMが単に次のサブゴールを出すだけでは不十分であり、現在の低レベル行動が目標達成に向かっているかを監視し、必要に応じて行動の選択傾向を補正する仕組みが必要になります。

そのため、本研究では、LLMによる計画生成に加えて、状態表現、視覚理解、記憶、失敗回復、低レベル方策への認知的介入、そして経験に基づく再学習を組み合わせたエージェントアーキテクチャを検討しています。

---

## 3. 研究の位置づけ

近年、Minecraftのようなオープンワールド環境において、LLMによる高レベル計画と、強化学習や模倣学習による低レベル行動制御を組み合わせる階層型エージェントの研究が進んでいます。

既存研究では、LLMがタスクをサブゴールやスキル列に分解し、低レベルポリシーが移動、採掘、クラフトなどの具体的な行動を実行する構成が多く見られます。

一方で、高レベルモジュールが一度サブゴールを決定した後、低レベルポリシーの実行中に、現在の行動が目標に向かっているかを継続的に監視し、必要に応じて補正・中断・再計画する仕組みは、まだ十分に扱われていないと考えています。

本研究では、この点に着目し、低レベル行動を単なる実行モジュールとして扱うのではなく、LLM/VLMが観察・判断・予測・記憶検索を通じて、実行中の行動に認知的に介入するアーキテクチャを目指しています。

さらに、介入を一時的な補正で終わらせるのではなく、介入によって得られた成功・失敗の経験を低レベル方策の再学習に利用します。これにより、最初は思考モジュールの補助が必要だった行動も、経験を重ねることで低レベル方策側に吸収され、より反射的・自律的に実行できるようになることを目指します。

これは、人間における「無意識的な運動制御」に対して、「意識的な注意や思考」が介入し、繰り返しの経験によって技能が身体化されていく仕組みに近い考え方です。

## 4. 主要なアイデア

本プロジェクトの中心的なアイデアは、以下の4つです。

* LLMによる高レベルな目標分解
* VLM・状態観測・記憶による現在状況の理解
* 低レベル行動の実行中に行う認知的介入
* 介入結果を経験として保存し、低レベル方策を再学習する仕組み

エージェントは、ユーザーから与えられた目標をサブゴール列へ分解し、各サブゴールを実行します。ただし、サブゴールを順番に実行するだけではなく、実行中の状態を観察し、現在の行動が目標に合っているかを判断します。

もし、対象物を見失った、移動が進んでいない、危険な状況になった、過去に似た失敗がある、現在のサブゴールが実行困難である、といった状況が発生した場合には、LLM/VLMベースの認知モジュールが行動の補正・中断・再計画を行います。

このとき、LLM/VLMは低レベル行動を毎ステップ直接決定するのではなく、低レベル方策が出す行動分布や行動選択の傾向に対して補助信号を与えることを想定しています。例えば、「もう少し左を見る」「対象ブロックに近づく」「現在の採掘対象を維持する」「危険なので後退する」といった形で、低レベル方策の出力を変調します。

その結果として成功した行動や、逆に失敗した行動は、エピソード記憶や手続き記憶として保存されます。そして、これらの経験を用いて低レベル方策を再学習することで、次回以降は思考モジュールによる介入なしでも、より適切な行動を選択できるようにすることを目指しています。

## 5. 認知的介入アーキテクチャ

本研究では、低レベル行動ポリシーやスキル実行に対して、LLM/VLMが実行中に介入できる構造を検討しています。

```text
ユーザーの目標
   ↓
LLM Planner
   ↓
Subgoal Queue
   ↓
Meta Controller
   ↓
Executor / Low-Level Policy
   ↓
Minecraft Agent
```

実行中は、Minecraft Agentから得られる状態情報をもとに、Cognitive Monitorが現在の行動を監視します。

```text
Minecraft Agent
   ↓
State Observer
   ├── Mineflayerから取得した記号的状態
   └── Minecraftスクリーンショットから取得した視覚的状態
   ↓
Cognitive Monitor
   ├── VLM: 現在の視覚的状況の理解
   ├── LLM: 目標との整合性判断
   ├── World Model: 行動結果の予測
   └── Memory System: 過去の類似経験の検索
   ↓
Intervention Controller
   ├── continue: 現在の行動を継続
   ├── modulate: 低レベル方策の行動分布を変調
   ├── correct: 視点・移動方向・探索方針を補正
   ├── interrupt: 現在の行動を中断
   └── replan: サブゴールを修正または再計画
   ↓
Executor / Low-Level Policy
   ↓
Experience Buffer
   ↓
Policy Relearning / Skill Update
```

この構成では、低レベルポリシーは移動、視点操作、採掘、攻撃、使用、クラフトなどの具体的な操作を担当します。一方で、LLM/VLMはそれらの操作をすべて直接決定するのではなく、実行中の状態を監視し、必要な場面で補助信号や修正方針を与えます。

例えば、低レベル方策が「木を切る」サブゴールを実行している最中に、視点が木から外れた場合、VLMが対象物の位置を認識し、LLMが現在のサブゴールとの整合性を判断します。その結果、Intervention Controllerは低レベル方策に対して、視点を戻す方向の行動を取りやすくする補助信号を与えます。

このように、LLM/VLMは低レベル方策を完全に置き換えるのではなく、状況に応じて行動分布を変調する役割を担います。

## 6. 行動分布の変調と再学習

本研究の重要なポイントは、LLM/VLMなどの思考モジュールが、低レベル方策を直接操作するのではなく、必要な場面で低レベル方策の行動分布を変調する点です。

低レベル方策は、現在の観測状態やサブゴールに基づいて、前進、後退、ジャンプ、視点移動、採掘、使用、クラフトなどの行動候補に対する確率分布を出力します。通常時は、この方策に従って行動を選択します。

しかし、以下のような場合には、思考モジュールが介入します。

* サブゴールの進行が止まっている
* 対象物を見失っている
* 視点や位置が目標とずれている
* 危険な状況に近づいている
* 過去に似た失敗が記憶されている
* 現在の方策だけでは繊細な動きが難しい
* World Modelが失敗確率の高さを予測している

このとき、LLM/VLMは「次の低レベル行動を完全に指定する」のではなく、低レベル方策の出力に対して補助的なバイアスを与えます。

例：

```text
Low-Level Policy:
  forward: 0.25
  left:    0.10
  right:   0.15
  jump:    0.05
  attack:  0.30
  look_up: 0.15

Cognitive Modulation:
  target tree is slightly left
  increase probability of left and attack
  decrease unrelated movement

Modulated Policy:
  forward: 0.20
  left:    0.25
  right:   0.05
  jump:    0.03
  attack:  0.40
  look_up: 0.07
```

このように、低レベル方策の出力を思考モジュールが状況に応じて調整することで、完全な手動制御よりも柔軟で、かつ毎ステップLLMに依存しない効率的な行動制御を目指します。

さらに、介入によって得られた行動系列と結果は、Experience Bufferに保存されます。

```text
状態
サブゴール
低レベル方策の元の行動分布
LLM/VLMによる変調内容
実際に選択された行動
成功・失敗
失敗理由
修復方法
```

これらのデータを用いて、低レベル方策を再学習します。これにより、最初はLLM/VLMによる補助が必要だった状況でも、次回以降は低レベル方策自身がより適切な行動分布を出力できるようになることを目指します。

この仕組みは、意識的な介入によって得られた経験が、繰り返しによって手続き記憶や反射的な行動へ移行していく過程に対応しています。

## 7. 記憶システム

本プロジェクトでは、人間の記憶の種類や機能を参考にして、エージェントの記憶システムを設計しています。

エージェントは、現在の状態や直近の行動を一時的に保持するだけでなく、重要な経験を整理し、長期的に再利用できる知識として保存することを目指しています。

```text
Memory System
├── Working Memory
│   └── 現在の目標・サブゴール・状態・直近の実行状況
│
├── Intermediate Memory
│   └── 直近の経験を一時的に保持し、重要な情報を長期記憶へ移行するための領域
│
└── Long-Term Memory
    ├── Declarative Memory
    │   ├── Episodic Memory
    │   │   └── 過去の経験、状態情報、失敗、反省など
    │   └── Semantic Memory
    │       └── クラフトレシピ、アイテム関係、タスク依存関係など
    │
    └── Non-Declarative Memory
        ├── Associative Learning
        ├── Non-Associative Learning
        ├── Priming
        ├── Procedural Memory
        └── Emotional Conditioning
```

特に本研究では、低レベル方策の再学習に関係する記憶として、Episodic MemoryとProcedural Memoryを重視しています。

Episodic Memoryには、どのような状態で、どのサブゴールを実行し、どのような失敗や成功が起きたかを保存します。Procedural Memoryには、特定の状況で有効だった行動パターンやスキルを保存し、低レベル方策の更新に利用します。

---

## 8. 中期記憶から長期記憶への移行

本プロジェクトでは、すべての経験をそのまま長期記憶へ保存するのではなく、一度中期記憶に保持し、重要なものだけを長期記憶へ移行する仕組みを検討しています。

長期記憶へ移行する候補には、以下のような情報があります。

* 同じ失敗が繰り返された経験
* 有効だった修復方法
* 他のタスクにも再利用できる知識
* タスク達成に重要だった視覚情報
* 危険や成功と強く結びついた状況
* 特定の行動と結果の関係
* LLM/VLMの介入によって改善された行動
* 低レベル方策の再学習に使える行動系列

```text
現在の状態・経験
   ↓
Working Memory
   ↓
Intermediate Memory
   ↓
重要度・新規性・再利用性・失敗頻度を評価
   ↓
Long-Term Memory
   ↓
Policy Relearning / Skill Update
```

このように、記憶システムは単なるログ保存ではなく、低レベル方策の改善や、失敗回復の再利用に接続されます。

## 9. 状態表現

これまでのバージョンでは、Mineflayerを用いてMinecraftから構造化された状態情報を取得していました。

例えば、インベントリ、位置情報、周囲のブロック、体力、現在のタスク、実行結果などを取得し、それらを計画、実行、失敗回復のための状態表現として利用していました。

一方で、Mineflayerによる状態表現は記号的な情報が中心であり、ゲーム画面全体の視覚的な状況を十分に表現することは難しいです。

そのため現在は、Minecraftのスクリーンショットから視覚情報を取得し、それを状態表現の一部として利用する方向に拡張しています。

状態表現には、以下のような情報を含めることを想定しています。

* インベントリ
* 位置情報
* 周囲のブロック
* 体力・空腹度
* 現在のサブゴール
* 実行中の低レベル行動
* 低レベル方策の行動分布
* 直近の進捗
* スクリーンショットから得られる視覚情報
* 過去の類似経験

---

## 10. 画像ベースの状態表現

画像ベースの状態表現は、以下のような用途に利用することを想定しています。

* 現在の環境をより詳細に理解する
* 対象物や障害物を視覚的に検出する
* 環境の変化を検出する
* サブゴールの実行を改善する
* 失敗時の反省を支援する
* 現在の計画がうまくいかない場合にサブゴールを修正する
* 低レベル方策の行動分布を補正する

テキストベースや記号ベースの状態情報だけに依存するのではなく、視覚情報を取り入れることで、エージェントがゲーム内の状況をより正確に理解できるようにすることを目指しています。

---

## 11. 非連合学習の応用

本研究では、非連合学習の一つである馴化の考え方を、画像ベースの状態理解に応用することを検討しています。

馴化とは、重要でない刺激が繰り返し与えられたときに、その刺激への反応や注意が徐々に小さくなる現象です。

Minecraftエージェントにおいては、繰り返し観測される重要度の低い視覚情報への注意を下げ、新規性が高い部分、不確実な部分、タスクに関係する部分へより注目する仕組みとして応用したいと考えています。

これにより、不要な観測コストを減らしながら、失敗時の反省、サブゴール修正、低レベル方策の補正に必要な情報へ集中できるようにすることを目指しています。

## 12. タスク例

入力目標：

```text

```

サブゴール列の例：

```text
find_tree
chop_tree
craft_planks
craft_crafting_table
craft_sticks
craft_wooden_pickaxe
mine_cobblestone
craft_stone_pickaxe
```

もしサブゴールの途中で失敗した場合、エージェントは現在の状態、過去の失敗事例、視覚的な状態表現、低レベル方策の行動分布を確認し、修復戦略を生成または再利用します。

例えば、木を切る途中で対象を見失った場合には、VLMが視覚的状況を確認し、LLMが目標との整合性を判断し、Memory Systemが過去の類似失敗を検索します。その結果に応じて、Intervention Controllerは視点補正、探索方針の変更、現在サブゴールの中断、別サブゴールへの再計画、または低レベル方策の行動分布の変調を行います。

そして、介入によって成功した場合、その状態・補正内容・行動系列・結果を保存し、低レベル方策の再学習に利用します。

## 14. 現在の実装状況

現在の実装では、Minecraftエージェントの構成として、主に2つのアプローチを実装しています。

### 14.1 Mineflayerベースのエージェント

1つ目は、Mineflayerを用いたエージェントです。

この構成では、Mineflayerを通じてMinecraft内の状態を取得し、LLM Plannerによってユーザーの高レベル目標を実行可能なサブゴール列へ分解します。

Executorは、生成されたサブゴールを順番に実行し、各サブゴールの成功・失敗、現在の状態、失敗理由などをエピソード記憶として保存します。

例えば、「石のツルハシを作る」という目標に対して、LLM Plannerが以下のようなサブゴール列を生成し、Mineflayerを用いてそれぞれの行動を実行します。

```text
find_tree
chop_tree
craft_planks
craft_crafting_table
craft_sticks
craft_wooden_pickaxe
mine_cobblestone
craft_stone_pickaxe
```

この実装では、インベントリ、位置情報、周囲のブロック、体力、現在のサブゴール、実行結果などを記号的な状態表現として扱っています。

また、実行中に失敗が発生した場合には、現在の状態や過去の失敗事例を参照し、サブゴールの修正や修復戦略の生成を行います。

### 14.2 STEVE-1ベースのエージェント

2つ目は、既存研究であるSTEVE-1を低レベル行動ポリシーとして利用するエージェントです。

STEVE-1は、Minecraftにおいてテキスト指示に基づいて低レベル行動を生成するモデルです。本プロジェクトでは、このSTEVE-1を低レベル処理として利用し、その上位にLLM Plannerを接続する構成を実装しました。

この構成では、ユーザーの高レベル目標をLLM Plannerがサブゴール列へ分解し、それぞれのサブゴールをテキスト指示としてSTEVE-1へ入力します。STEVE-1は、その指示に基づいてMinecraft内で低レベルの行動を実行します。

```text
ユーザーの目標
   ↓
LLM Planner
   ↓
Subgoal Queue
   ↓
Text Instruction
   ↓
STEVE-1
   ↓
Minecraft内の低レベル行動
```

例えば、「石のツルハシを作る」という目標に対して、LLM Plannerが「木を探す」「木を切る」「板材を作る」「棒を作る」「丸石を掘る」などのサブゴールを生成し、それぞれをSTEVE-1へ入力することで、低レベル行動の実行まで行います。

この実装により、MineflayerのAPIベースの行動実行だけでなく、視覚入力とテキスト指示に基づく低レベル行動ポリシーを用いたエージェント構成も検証しています。

今後は、この低レベル方策に対して、LLM/VLMによる行動分布の変調や、介入結果を用いた再学習の仕組みを統合していく予定です。
